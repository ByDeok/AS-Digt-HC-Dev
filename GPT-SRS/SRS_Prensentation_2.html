SRS v0.2 요약

액티브시니어·보호자용 디지털 헬스케어 상호운용 허브 – Software Requirements Specification
  • 문서 ID SRS-001 / Revision 1.0 / 2025-11-15
  • 표준 ISO/IEC/IEEE 29148:2018 기반 SRS
  • Source of Truth GPT-PRD/GPT-PRD.md 단일 PRD를 100% 기반으로 작성

목적: PRD의 비전·Story·KPI·기능·NFR을 테스트 가능하고 추적 가능한 요구사항(REQ-FUNC / REQ-NF)으로 구조화.

1. 비전 & 범위

1-1. 제품 비전 (PRD 인용)

“시니어·보호자 이중 UX와 의료급 신뢰 포맷을 갖춘 상호운용 허브로, 3분 온보딩에서 시작해 의사용 1장 요약과 ‘오늘 3가지 행동’을 통해 진료 수용성과 일상 순응을 동시에 향상한다.”

1-2. In-Scope (핵심 기능)

  • F1 – 3분 온보딩 일괄 흐름: 가입·인증·권한·디바이스/병원 연동
  • F2 – 1장 의사용 요약 리포트: 3~6개월 데이터 요약, PDF·EMR 첨부
  • F3 – ‘오늘 1~3개 행동’ 카드: 일일 행동 코칭 및 완료 추적
  • F4 – 가족 보드 / 대리 접근, F5 – 병원 포털 1곳 연동, F7 – 의사 신뢰 포맷
  • F8 – 권한·프라이버시 단순화, F10 – 접근성 퍼스트(A11y)

1-3. Out-of-Scope (초기 버전 제외)

  • 완전한 EMR 양방향 연동, 보험 청구/심사, 의료진 대시보드 상용 출시
  • 고급 개인화 AI 코치, 글로벌 출시 및 각국 규제 최적화

2. 이해관계자 & 시스템 컨텍스트

2-1. Stakeholders

  • STK-01 시니어: 데이터 기록·조회, 리포트 생성, 행동 수행
  • STK-02 보호자: 가족 보드에서 일정·복약·위험 알림 관리
  • STK-03 의사: 진료 시 1장 리포트 열람·EMR 첨부
  • STK-04 병원 관리자: 포털/EMR 연동, 파일럿 성과 관리
  • STK-06 DevOps: SLA, 오류율, 인프라 비용 관리

2-2. System Context

  • EX-01 Hospital Portal / EMR
  • EX-02 Device SDKs (워치, 혈압계 등)
  • EX-03 Auth Provider (소셜 / 휴대폰)
  • EX-04 Notification Service

클라이언트는 Mobile App 중심, Web Console은 보호자/병원용 보조 채널로 정의.

3. 핵심 시퀀스 요약

3-1. 3분 온보딩 플로우

Senior(App) → Auth: 로그인/본인인증
Senior → Backend: 프로필 + 동의 정보 전송
Senior → Device SDK: OAuth 연동, 초기 데이터 동기화
Senior → Hospital Portal: (지원 시) 포털 연동 및 검사 결과 수신
Backend → Senior: p50 ≤ 180초 내 온보딩 완료, 첫 리포트/행동 카드 진입

3-2. 진료 전 1장 리포트 & 연동 상태 동기화

  • 리포트 생성: 최신 검사 결과 + 활동/심박/혈압/체중 집계 → 1장 요약
  • 현장 사용: 앱/PDF로 제시, 3클릭 이내 EMR 첨부
  • 연동 상태: DeviceLink / PortalConnection 테이블로 토큰·포털 상태 주기 점검

4. Functional Requirements 구조

4-1. Story → REQ-FUNC 분해

  • Story 1 1장 요약 리포트 → REQ-FUNC-007~010
  • Story 2 오늘 1~3개 행동 → REQ-FUNC-011~014
  • Story 3 가족 보드/역할 → REQ-FUNC-015~018
  • Story 4 3분 온보딩 → REQ-FUNC-001~006,019

4-2. 예시 (온보딩 관련)

  • REQ-FUNC-001: 신규 시니어 프로필 생성(나이/성별/질환)
  • REQ-FUNC-003: 최소 2종 디바이스(OAuth 연동 + 초기 동기화)
  • REQ-FUNC-006: 필수 단계 완료 시 “온보딩 완료” 판정 및 첫 가치 제공

각 REQ-FUNC는 Story와 Acceptance Criteria를 그대로 가져와 테스트 가능하게 서술.

5. Non-Functional Requirements & KPI

5-1. 성능·품질 지표

  • REQ-NF-001: 앱 초기 로드 p95 ≤ 1.5s, 화면 전환 p95 ≤ 800ms
  • REQ-NF-002: 리포트 생성 p95 ≤ 3s, PDF 렌더링 p95 ≤ 2s
  • REQ-NF-003: 온보딩 p50 ≤ 180s, 성공률 ≥ 65%
  • REQ-NF-004: 월 가용성 ≥ 99.5%, 오류율 < 0.5%
  • REQ-NF-008: 접근성 – 라벨 누락/포커스 트랩 0건

5-2. KPI & Go-to-Market

  • REQ-NF-011: 리포트 채택률 10% → 40%+
  • REQ-NF-012: 온보딩 완료율 35% → 65%+
  • REQ-NF-013: 월간 리포트 50%+, 대리접근 50%+
  • REQ-NF-014: 90일 유지율 35%+
  • REQ-NF-019: 기관 채널 3곳+ MoU, 커넥터 4종 활성 (Go-to-Market)

6. 데이터 모델 & Traceability

6-1. 핵심 엔터티

  • User/Profile, OnboardingSession, HealthReport, ActionCard, FamilyBoard/AccessRole
  • 보안/연동 ConsentRecord, AuditLog, DeviceLink, PortalConnection

6-2. Traceability Matrix

  • 각 Story별로 REQ-FUNC / REQ-NF / Test Case ID(TC-Sx-yy) 양방향 매핑
  • 예: Story 1 → REQ-FUNC-007~010, REQ-NF-002,008,011,016 → TC-S1-01~06

결과적으로 PRD Story → SRS 요구사항 → 테스트 케이스까지 end-to-end 추적 가능.
